# Degeneration Axis

**Analyzing and mitigating LLM activation drift in agentic trajectories**

## Motivation

When LLMs operate as agents over long trajectories (50–100+ steps), do their internal activations drift systematically? If so, can we detect that drift and mitigate it with inference-time steering?

This project investigates activation drift on [SWE-bench](https://www.swebench.com/) trajectories generated by [mini-swe-agent](https://mini-swe-agent.com/).

### Supported models

| Model | Params | Layers | Hidden dim | Type |
|-------|--------|--------|------------|------|
| [Qwen3-4B](https://huggingface.co/Qwen/Qwen3-4B) | 4B | 36 | 2560 | Dense |
| [Qwen3.5-9B](https://huggingface.co/Qwen/Qwen3.5-9B) | 9B | 32 | 4096 | Hybrid (DeltaNet + Attention) |
| [Qwen3.5-27B](https://huggingface.co/Qwen/Qwen3.5-27B) | 27B | 64 | 5120 | Hybrid (DeltaNet + Attention) |
| [Qwen3.5-35B-A3B](https://huggingface.co/Qwen/Qwen3.5-35B-A3B) | 35B (3B active) | 40 | 2048 | Hybrid MoE (DeltaNet + Attention + MoE) |

The key insight: rather than detecting *temporal* drift via PCA, we use an **LLM judge** to label steps as good or bad, then compute a **contrastive** drift axis: `mean(bad) - mean(good)`.

## Pipeline

The experiment is a 6-step pipeline (steps 0–5). Each step reads from the previous step's output.

```
vLLM server → 16 trajectories/problem → LLM judge labels → vLLM activations → contrastive axis → steering
     0                  1                       2                  3                  4              5
```

### Step 0: Start vLLM server

Serves the target model locally with an OpenAI-compatible API for trajectory collection.

```bash
./pipeline/0_vllm_server_qwen3_4b.sh       # Qwen3-4B
./pipeline/0_vllm_server_qwen35_9b.sh      # Qwen3.5-9B
./pipeline/0_vllm_server_qwen35_27b.sh     # Qwen3.5-27B
./pipeline/0_vllm_server_qwen35_35b.sh     # Qwen3.5-35B-A3B
```

### Step 1: Collect trajectories

Runs mini-swe-agent 16 times per problem (temperature=0.7) on SWE-bench verified split using GNU Parallel. After collection, reorganizes files into a problem-based layout.

```bash
./pipeline/1_collect_trajectories.sh [WORKERS_PER_RUN] [PARALLEL_RUNS]
```

Output: `data/drift/qwen3-4b/raw/{instance_id}/run_00.traj.json ... run_15.traj.json`

### Step 2: LLM-as-Judge labeling

Evaluates each assistant turn on hallucination and dishonesty using GPT-5.2 via OpenRouter/litellm.

```bash
python pipeline/2_judge_steps.py
```

Output: `data/drift/qwen3-4b/judgments/{instance_id}__run_{RUN_ID}.judgments.json`

### Step 3: Extract per-step activations

Single forward pass per trajectory via `VllmHiddenStatesGenerator`. Tokenizes the full conversation, extracts hidden states, then slices and mean-pools per assistant turn span.

```bash
python pipeline/3_activations.py
```

Output: `data/drift/qwen3-4b/activations/{instance_id}__run_{RUN_ID}.pt` — tensors of shape `(N_steps, 10, 2560)`

### Step 4: Compute contrastive drift axis

Joins activations with judge labels. For each problem, computes `mean(bad) - mean(good)`, then averages across problems. This controls for problem difficulty.

```bash
python pipeline/4_compute_axis.py --metric hallucination
```

Output: `data/drift/qwen3-4b/analysis/drift_axis.pt` and diagnostic plots

### Step 5: Drift steering

Caps activations along the contrastive axis. Thresholds are derived from good steps (label=0): `threshold = mean(good_projections) + n_std * std(good_projections)`.

```bash
python pipeline/5_drift_steering.py
```

## Key concepts

**Contrastive axis**: Instead of PCA-based temporal drift detection, we compute `mean(bad_activations) - mean(good_activations)` where good/bad labels come from an LLM judge. Per-problem averaging controls for difficulty.

**LLM-as-Judge**: GPT-5.2 evaluates each assistant step on hallucination (fabricated info) and dishonesty (misrepresented progress). Labels: 0=good, 1=bad, 2=ambiguous.

**Single-pass extraction**: For causal models, a single forward pass over the full conversation gives correct activations for all positions (position p attends to 0..p). This replaces the N incremental passes from the old pipeline.

**Activation capping**: A steering intervention that clamps the projection of activations onto the drift axis at a threshold derived from good steps. Projections below the threshold are untouched; those above are pulled back.
